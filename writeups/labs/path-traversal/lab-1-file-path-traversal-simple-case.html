Lab 1 — File path traversal, simple case

🔴 High 📅 2026-07-12

📋 Finding Summary

FieldValue
TypeWeb App
Severity🔴 High
CVE / CWECWE-22: Improper Limitation of a Pathname to a Restricted Directory
Date2026-07-12

🔍 Description

The application loads product images by passing a user-supplied filename parameter directly to a server-side file read in the /image endpoint, with no sanitization or validation of path traversal sequences. By replacing the legitimate filename with ../../../etc/passwd, an attacker can traverse three directory levels above the web root and instruct the server to return the contents of arbitrary files. This is the simplest form of path traversal — no filter bypass is required.

🎯 End Goals

  • Exploit the path traversal vulnerability in the product image endpoint
  • Retrieve the contents of /etc/passwd from the server

🪜 Steps to Reproduce

  1. Browse to any product page on the shop to trigger product image requests
  2. Intercept the image request in Burp Proxy and identify the filename parameter
  3. Send the request to Repeater and modify the filename parameter to ../../../etc/passwd
  4. Send the request and observe a 200 OK response returning the contents of /etc/passwd

🧠 Analysis

Step 1 — Browse the shop

Opened the lab environment and browsed to the shop homepage. The lab banner confirmed the title “File path traversal, simple case” with status “Not solved”.

Step 2 — Capture image request in Burp HTTP history

With Burp Proxy active, browsed the shop to trigger product image loads. The HTTP history showed GET requests to /image?filename=36.jpg, exposing the filename parameter used to serve product images.

Step 3 — Test absolute path (fails)

Sent the image request to Repeater and modified the filename parameter to /etc/passwd. The server returned 400 Bad Request with "No such file", indicating the absolute path attempt failed — the server resolves paths relative to its image directory.

Step 4 — Inject path traversal payload (200 OK)

Modified the filename parameter to ../../../etc/passwd. The three ../ sequences traverse three directory levels up from the images directory to the filesystem root, then descend into /etc/passwd. The server returned 200 OK with Content-Type image/jpeg and the full contents of /etc/passwd in the response body — confirming no sanitization is applied to the filename parameter.

Step 5 — Verify lab solved in browser

Navigated back to the lab in the browser, which displayed the “Congratulations, you solved the lab!” banner confirming successful exploitation.

🎯 Impact

An attacker with access to the image endpoint can read any file on the server that the web process has permission to access. This includes sensitive system files such as /etc/passwd, /etc/shadow, application configuration files containing database credentials, private SSH keys, and source code. No authentication or special privileges are required — the endpoint is publicly accessible.
