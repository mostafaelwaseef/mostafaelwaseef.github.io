Lab 6 — File path traversal, validation of file extension with null byte bypass

🔴 High 📅 2026-07-12

📋 Finding Summary

FieldValue
TypeWeb App
Severity🔴 High
CVE / CWECWE-22: Improper Limitation of a Pathname to a Restricted Directory
Date2026-07-12

🔍 Description

The application validates that the user-supplied filename parameter in the /image endpoint ends with an expected file extension (e.g. .png). However, the extension check is performed on the raw string before the server processes path components, and the server does not strip or reject null byte characters (%00). By appending a null byte followed by a valid extension — for example, ../../../etc/passwd%00.png — the extension check sees .png at the end and passes, while the underlying file-access layer treats the null byte as a string terminator, reading only ../../../etc/passwd and returning the contents of that file.

🎯 End Goals

  • Exploit the path traversal vulnerability in the product image endpoint
  • Bypass the file extension validation using a null byte injection
  • Retrieve the contents of /etc/passwd from the server

🪜 Steps to Reproduce

  1. Browse to any product page on the shop to trigger product image requests
  2. Intercept an image request in Burp Proxy and identify the filename parameter
  3. Send the request to Repeater and confirm that the server blocks traversal and extension-less payloads
  4. Modify the filename parameter to ../../../etc/passwd%00.png to satisfy the extension check while terminating the path at the null byte
  5. Send the request and observe a 200 OK response returning the contents of /etc/passwd

🧠 Analysis

Step 1 — Browse the shop

Opened the lab environment and browsed to the shop homepage. The lab banner confirmed the title “File path traversal, validation of file extension with null byte bypass” with status “Not solved”.

Step 2 — Capture image request in Burp HTTP history

With Burp Proxy active, browsed the shop to trigger product image loads. The HTTP history showed GET requests to /image?filename=26.jpg, exposing the filename parameter used to serve product images.

Step 3 — Test multiple payloads (all fail)

Sent the image request to Repeater and tested five payloads to probe the validation logic. /etc/passwd (absolute path) returned 400 “No such file”. ../../../etc/passwd (plain traversal) also returned 400. ....//....//....//etc/passwd (non-recursive bypass) returned 400. Single URL-encoded %2e%2e%2f%2e%2e%2f%2e%2e%2f%65%74%63%2f%70%61%73%73%77%64 returned 400. A double URL-encoded variant also returned 400. All attempts failed — the server enforces an extension check that rejects paths not ending with the expected file extension.

Step 4 — Inject null byte bypass payload (200 OK)

Modified the filename parameter to ../../../etc/passwd%00.png. The null byte (%00) acts as a string terminator at the OS level: the extension validator reads the full string and sees .png at the end, satisfying the check. When the server passes the path to the filesystem, the null byte terminates the string at ../../../etc/passwd, causing the server to read and return the contents of /etc/passwd. The response returned 200 OK with Content-Type image/png and the full /etc/passwd contents in the response body.

Step 5 — Verify lab solved in browser

Navigated back to the lab in the browser, which displayed the “Congratulations, you solved the lab!” banner confirming successful exploitation.

🎯 Impact

An attacker can bypass file extension validation by injecting a null byte before the required extension suffix. Any server that validates the extension on the raw input string without also sanitizing null bytes — and whose file-access layer respects null byte string termination — is vulnerable. Sensitive files such as /etc/passwd, /etc/shadow, application configuration files, private keys, and source code are all at risk of unauthorized disclosure.
