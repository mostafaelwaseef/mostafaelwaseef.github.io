Lab 3 — DOM XSS in document.write sink using source location.search

🟡 Medium 📅 2026-05-20

📋 Finding Summary

FieldValue
TypeWeb App
Severity🟡 Medium
CVE / CWECWE-79
Date2026-05-20

🔍 Description

DOM-based Cross-Site Scripting occurs when client-side JavaScript reads attacker-controllable data from the DOM (the source) and writes it to a dangerous sink without sanitization. In this lab, the blog search functionality uses document.write() as the sink and location.search as the source. The trackSearch() function reads the search query directly from the URL and writes an <img> tag containing the unsanitized value into the page DOM. Because the input is not encoded before being written, an attacker can break out of the src attribute context and inject arbitrary event handler attributes that execute JavaScript.

🎯 End Goals

  • Identify the vulnerable source (location.search) and sink (document.write)
  • Craft a payload that breaks out of the injected <img> tag’s attribute and calls alert
  • Confirm JavaScript execution via the alert dialog
  • Solve the lab

🪜 Steps to Reproduce

  1. Access the lab and perform a test search to observe how input is reflected
  2. Inspect the page source to identify the document.write sink and location.search source
  3. Craft a payload that breaks out of the src attribute of the injected <img> tag and appends an event handler
  4. Submit the payload via the search field and confirm the alert fires
  5. Verify the lab is marked as solved

🧠 Analysis

Step 1 — Access the lab
The lab environment loaded a blog application with a search field. The lab status showed “Not solved”.

Step 2 — Test with a benign search term
The term hello was entered into the search box and submitted. The page reflected the search term in the results heading, indicating the input flows from location.search into the page output.

Step 3 — Inspect the page source
DevTools was opened to inspect the rendered HTML. The JavaScript trackSearch() function was found calling document.write() with data sourced directly from location.search. It writes an <img> tag with the search term placed unsanitized inside the src attribute — for example: <img src="/resources/images/tracker.gif?searchTerms=hello">. This confirms document.write as the sink and location.search as the controllable source.

Step 4 — Craft the XSS payload
The payload hello" onload="alert('hacked') was entered into the search field. This payload closes the src attribute of the <img> tag with a double quote, then injects an onload event handler. When document.write renders this into the DOM, the resulting tag becomes <img src="/resources/images/tracker.gif?searchTerms=hello" onload="alert('hacked')">, which executes when the image loads.

Step 5 — alert() fires
Upon page load, the injected onload handler executed and an alert dialog appeared with the text “hacked”, confirming successful DOM XSS exploitation via the document.write sink.

Step 6 — Lab solved
After dismissing the alert, the page displayed “Congratulations, you solved the lab!” and the lab status updated to Solved.

🎯 Impact

An attacker can craft a malicious URL containing the XSS payload in the search query parameter and trick a victim into clicking it. The payload executes entirely client-side without any server interaction, making it harder to detect via server-side logging. Exploitation can lead to session hijacking, credential theft, malicious redirects, or keylogging — all within the context of the trusted domain.
