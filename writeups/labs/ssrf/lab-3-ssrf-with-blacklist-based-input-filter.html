Lab 3 — SSRF with blacklist-based input filter

⚪ 🟠 High 📅 2026-07-03

📋 Finding Summary

FieldValue
TypeWeb App
Severity🟠 High
CVE / CWECWE-918
Date2026-07-03

🔍 Description

Server-Side Request Forgery (SSRF) is a vulnerability that allows an attacker to induce a server-side application to make HTTP requests to an arbitrary destination of the attacker’s choosing. The application’s stock-check feature accepts a stockApi parameter containing the full URL the back-end server should fetch to retrieve stock data. The developer has deployed a blacklist that rejects requests containing obvious loopback references (localhost, 127.0.0.1) and the literal string admin, but blacklist-based filtering is inherently incomplete: it can be bypassed using alternate representations of the same address (such as the shorthand loopback notation 127.1) and using double URL-encoding, since the string comparison happens before the value is fully decoded but the underlying HTTP client still resolves the doubly-encoded characters before making the request.

🎯 End Goals

  • Identify the stock-check feature and confirm the stockApi parameter is server-side fetched
  • Bypass the blacklist’s loopback-address filtering using an alternate address representation
  • Bypass the blacklist’s admin keyword filtering using double URL-encoding
  • Delete the user carlos via the admin interface to solve the lab

🪜 Steps to Reproduce

  1. Open the lab and browse the shop homepage.
  2. Navigate to a product page and use Burp Suite to intercept the stock-check request.
  3. Identify the stockApi parameter and confirm it is fetched server-side.
  4. Attempt obvious loopback references and observe that the blacklist blocks them.
  5. Bypass the loopback filter using an alternate address representation.
  6. Attempt to reach the /admin path and observe that the blacklist blocks the keyword.
  7. Bypass the keyword filter using double URL-encoding.
  8. Enumerate the admin panel’s users and inspect the delete link’s structure.
  9. Route the delete request through the stockApi parameter using both bypasses together.
  10. Confirm the lab is solved and the target user has been deleted.

🧠 Analysis

Step 1 — Access the lab

The lab was launched and the shop homepage loaded successfully.

Step 2 — Locate the stock-check feature and capture the baseline request

Navigating to a product page revealed a stock-check feature. With Burp Suite intercepting traffic, submitting the stock check captured a request containing a stockApi parameter with the URL-encoded value http://stock.weliketoshop.net:8080/product/stock/check?productId=1&storeId=1. The response returned HTTP/2 200 with a plain-text body of 985.

Step 3 — Inconclusive probe

The stockApi value was truncated to http://stock.weliketoshop.net:8080/, with no path or query string. The response was HTTP/2 400 Bad Request with the body "Missing parameter".

Step 4 — Obvious loopback references are blocked

Setting stockApi to http://localhost/ returned HTTP/2 400 Bad Request with the body "External stock check blocked for security reasons". Trying http://127.0.0.1/ produced the identical blocked response, confirming the developer’s blacklist explicitly matches both of these common loopback references.

Step 5 — Bypass the blacklist via an alternate loopback representation

Setting stockApi to http://127.1/ — a valid but less common shorthand notation for 127.0.0.1 — returned HTTP/2 200 OK with the application’s own homepage HTML, bypassing the blacklist entirely since it never accounted for this alternate address form. The browser’s navigation bar subsequently showed a new “Admin panel” link.

Step 6 — The /admin path is blocked again

Extending the request to http://127.1/admin/ returned HTTP/2 400 Bad Request with the same "External stock check blocked for security reasons" message, showing the blacklist also matches the literal string admin anywhere in the URL.

Step 7 — Bypass the “admin” keyword via double URL-encoding

Single URL-encoding the letter a in admin (%61dmin) still returned the blocked response, since the application appears to decode the value once before running the blacklist check. Encoding it a second time — %2561dmin, where %25 is the encoded form of the percent sign itself — succeeded: stockApi=http://127.1/%2561dmin/ returned HTTP/2 200 OK with the admin panel’s HTML, titled “SSRF with blacklist-based input filter.” The double-encoded value passes the blacklist check as a literal string, then gets decoded to admin when the server actually issues the request.

Step 8 — Enumerate users and inspect the delete link

Following the admin panel link displayed a “Users” page listing two accounts, wiener and carlos, each with a “Delete” link. Inspecting the element in DevTools showed the underlying href was a plain relative path, /admin/delete?username=carlos.

Step 9 — Exploit via SSRF to delete carlos

The stockApi parameter was set to http://127.1/%2561dmin/delete?username=carlos, combining both bypasses, and sent through Repeater. The response was HTTP/2 302 Found with a Location: /admin redirect, indicating the deletion request succeeded when routed server-side through the stockApi parameter.

Step 10 — Verify and solve

Navigating directly to /admin in the browser still returned “Admin interface only available if logged in as an administrator, or if requested from loopback,” confirming the loopback restriction remains enforced for ordinary requests even after the lab was solved. A final refresh of the product page showed “User deleted successfully!” with the Users list now containing only wiener, confirming carlos had been removed via the SSRF exploit.

🎯 Impact

Blacklist-based SSRF defenses are fundamentally fragile because they attempt to enumerate every dangerous input rather than restrict to a known-safe set. In this lab, that fragility allowed an unauthenticated attacker to reach an internal admin interface by using an alternate address notation the developer never anticipated, and to bypass a keyword filter entirely using double URL-encoding — ultimately performing a destructive action (deleting a user account) that the application’s own access controls were explicitly designed to prevent from anywhere but the loopback interface. In a real deployment, the same class of bypass could expose any internal service the blacklist failed to account for.
