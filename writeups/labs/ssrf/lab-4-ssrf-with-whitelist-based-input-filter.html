Lab 4 — SSRF with whitelist-based input filter

🔴 High 📅 2026-07-03

📋 Finding Summary

FieldValue
TypeWeb App
Severity🟠 High
CVE / CWECWE-918
Date2026-07-03

🔍 Description

Server-Side Request Forgery (SSRF) is a vulnerability that allows an attacker to induce a server-side application to make HTTP requests to an arbitrary destination of the attacker’s choosing. The application’s stock-check feature accepts a stockApi parameter containing the full URL the back-end server should fetch to retrieve stock data. In this variant, the developer has deployed a whitelist that requires the URL’s hostname to exactly match stock.weliketoshop.net. However, the validating code and the code that actually issues the request parse the URL differently: embedding credentials (user@host syntax) reveals that the real connection attempt targets whatever precedes the @, and double URL-encoding the # fragment character lets an attacker smuggle a second hostname past the whitelist check entirely, since the validator only sees a single decode pass while the underlying HTTP client decodes it twice.

🎯 End Goals

  • Identify the stock-check feature and confirm the stockApi parameter is server-side fetched
  • Discover the parser discrepancy exposed by embedded URL credentials
  • Bypass the hostname whitelist using double URL-encoding of the # fragment character
  • Delete the user carlos via the admin interface to solve the lab

🪜 Steps to Reproduce

  1. Open the lab and browse the shop homepage.
  2. Navigate to a product page and use Burp Suite to intercept the stock-check request.
  3. Identify the stockApi parameter and confirm it is fetched server-side.
  4. Attempt an obvious loopback reference and observe that the whitelist blocks it.
  5. Embed credentials in the URL (user@host) and observe the whitelist passes it, but the connection fails.
  6. Single URL-encode the # character and observe the whitelist still blocks it.
  7. Double URL-encode the # character to bypass the whitelist.
  8. Substitute localhost for the placeholder credential to reach the local server.
  9. Enumerate the admin panel’s users and inspect the delete link’s structure.
  10. Route the delete request through the stockApi parameter using the bypass.
  11. Confirm the lab is solved and the target user has been deleted.

🧠 Analysis

Step 1 — Access the lab

The lab was launched and the shop homepage loaded successfully.

Step 2 — Locate the stock-check feature and capture the baseline request

Navigating to a product page revealed a stock-check feature. With Burp Suite intercepting traffic, submitting the stock check captured a request containing a stockApi parameter with the URL-encoded value http://stock.weliketoshop.net:8080/product/stock/check?productId=1&storeId=1. The response returned HTTP/2 200 with a plain-text body of 387.

Step 3 — Inconclusive probe

The stockApi value was truncated, with no path or query string. The response was HTTP/2 400 Bad Request with the body "Missing parameter".

Step 4 — Obvious loopback reference is blocked by the whitelist

Setting stockApi to http://localhost/ returned HTTP/2 400 Bad Request with the body "External stock check host must be stock.weliketoshop.net", confirming the whitelist validates the hostname exactly rather than using a blacklist of forbidden values.

Step 5 — Discover the embedded-credentials parser discrepancy

Setting stockApi to http://username@stock.weliketoshop.net/ passed the whitelist check, since the hostname portion the validator reads is still stock.weliketoshop.net. However, the response returned HTTP/2 500 with the body "Could not connect to external stock check service", indicating the code that actually issues the request instead tried to connect to username — revealing that the validating logic and the requesting logic parse the URL’s host differently.

Step 6 — Confirm single-encoding the # is still filtered

Attempting http://username%23@stock.weliketoshop.net — single URL-encoding the # character — still returned HTTP/2 400 Bad Request with the whitelist’s blocking message, since the value is decoded once before the hostname check runs.

Step 7 — Bypass the whitelist via double URL-encoding

Double-encoding the # character as %2523 — http://username%2523@stock.weliketoshop.net — passed the whitelist check, since the validator only decodes it once and still sees stock.weliketoshop.net as the host. The response returned HTTP/2 500 with the same "Could not connect to external stock check service" message as Step 5, confirming the requesting code fully decodes the value twice, treats everything after the real # as a URL fragment, and attempts to connect to username instead — now unblocked by the whitelist entirely.

Step 8 — Prove SSRF against the local server

Substituting localhost for the placeholder credential — http://localhost%2523@stock.weliketoshop.net — returned HTTP/2 200 OK with the application’s own homepage HTML, confirming the request actually reached the local server. The browser’s navigation bar subsequently showed a new “Admin panel” link.

Step 9 — Enumerate users and inspect the delete link

Following the admin panel link via the same bypass (.../admin) returned HTTP/2 200 OK with the admin panel’s HTML. The Users page listed two accounts, wiener and carlos, each with a “Delete” link. Inspecting the element in DevTools showed the underlying href was /admin/delete?username=carlos.

Step 10 — Exploit via SSRF to delete carlos

The stockApi parameter was set to http://localhost%2523@stock.weliketoshop.net/admin/delete?username=carlos and sent through Repeater. The response was HTTP/2 302 Found with a Location: /admin redirect, indicating the deletion request succeeded when routed server-side through the stockApi parameter.

Step 11 — Verify and solve

Navigating directly to /admin in the browser still returned the whitelist/loopback-style access restriction, confirming the block remained in place for ordinary requests even after the lab was solved. The admin panel was re-confirmed reachable via the double-encoding bypass, and a final refresh of the product page showed “User deleted successfully!” with the Users list now containing only wiener, confirming carlos had been removed via the SSRF exploit.

🎯 Impact

Whitelist-based SSRF defenses are only as strong as the consistency between the code that validates a URL and the code that ultimately requests it. In this lab, that inconsistency allowed an unauthenticated attacker to pass a strict hostname whitelist by embedding a second hostname as URL credentials, and to fully defeat the filter using double URL-encoding of the # fragment character — ultimately performing a destructive action (deleting a user account) against an admin interface that was never meant to be reachable externally. In a real deployment, the same class of parser discrepancy could expose any internal service behind a whitelist that assumes a single, uniform URL-parsing behavior across its own components.
