Web Shell Upload via Content-Type Restriction Bypass

⚪ 🔴 High 📅 2026-05-13

📋 Finding Summary

FieldValue
TypeWeb App
TargetPortSwigger Web Security Academy
Severity🔴 High
CVE / CWECWE-434: Unrestricted Upload of File with Dangerous Type
Date2026-05-13

🔍 Description

The application attempts to restrict file uploads by checking the Content-Type header of the request. However, since this header is user-controllable, an attacker can intercept the upload request and modify the Content-Type value to an allowed type (e.g. image/jpeg) while uploading a malicious PHP web shell. The server trusts this header without performing any server-side validation of the actual file content, allowing the PHP file to be stored and subsequently executed.

🎯 End Goals

  • Bypass the Content-Type restriction to upload a PHP web shell
  • Use the uploaded web shell to read the contents of /home/carlos/secret
  • Submit the secret to solve the lab

🪜 Steps to Reproduce

  1. Log in using the provided credentials (wiener:peter)
  2. Navigate to the account page and upload a benign image to confirm the upload function works
  3. Attempt to upload a PHP web shell directly — observe the server rejects it based on Content-Type
  4. Intercept the PHP upload request in Burp Suite and change the Content-Type header from application/x-php to image/jpeg
  5. Forward the modified request and confirm the upload succeeds
  6. Send a GET request to the uploaded PHP file's path to trigger execution and retrieve the secret
  7. Submit the secret to solve the lab

🧠 Analysis

Step 1 — Logging in and observing the upload function

Logged in as wiener using credentials wiener:peter. The account page presented an avatar upload form. A cat image was already set as the avatar, confirming the upload function was previously used.

Step 2 — Uploading a benign image to confirm behavior

Selected images.jpeg and uploaded it via the avatar form. The server accepted it and responded: The file avatars/images.jpeg has been uploaded. Burp Suite confirmed the POST request to /my-account/avatar returned HTTP 200 OK with Content-Type: text/html.

Step 3 — Crafting the payload and attempting to upload the PHP web shell directly

The PHP web shell exploit.php was created using nano with the following payload:

<?php echo file_get_contents('/home/carlos/secret'); ?>

The file was then uploaded directly without modification. The server rejected it with the error: Sorry, file type application/x-php is not allowed. Only image/jpeg and image/png are allowed. Burp confirmed the POST returned HTTP 403 Forbidden, revealing the server validates only the Content-Type header.

Step 4 — Bypassing the restriction via Content-Type manipulation

With Burp Suite Intercept enabled, the PHP upload request was captured. The Content-Type header in the multipart form data was changed from application/x-php to image/jpeg while keeping the PHP payload intact. The modified request was forwarded. The server accepted it and responded: The file avatars/exploit.php has been uploaded.

Step 5 — Triggering the web shell and retrieving the secret

A GET request was sent directly to /files/avatars/exploit.php. The server executed the PHP file and returned the secret:

kFOZfjRYDMFRmMNmuDmS0wn6sVJXVl2

The secret was submitted via the lab banner to confirm the lab was solved.

🎯 Impact

An attacker can bypass Content-Type-based upload restrictions by manipulating the request header, upload a PHP web shell, and achieve Remote Code Execution on the server. This allows reading of sensitive files, data exfiltration, and potential full server compromise.
