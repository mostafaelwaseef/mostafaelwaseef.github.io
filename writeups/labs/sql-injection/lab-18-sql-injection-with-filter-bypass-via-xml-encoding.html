Lab 18 — SQL injection with filter bypass via XML encoding

🔴 High 📅 2026-05-19

📋 Finding Summary

FieldValue
TypeWeb App
Severity🔴 High
CVE / CWECWE-89
Date2026-05-19

🔍 Description

SQL injection with filter bypass via XML encoding occurs when a web application passes user-supplied data embedded in an XML body to a SQL query, and a WAF or input filter is in place to block common SQL injection keywords such as UNION and SELECT. An attacker can evade the filter by obfuscating the payload using XML entity encoding — for example, representing each character as its hex entity equivalent (&#x55;&#x4e;&#x49;&#x4f;&#x4e;...). The database’s XML parser decodes the entities before the query is executed, so the SQL runs normally while the WAF never sees the plaintext keywords. In this lab, the stock check feature sends a POST request with an XML body containing productId and storeId values. The storeId field is vulnerable to SQL injection, and the Hackvertor Burp extension is used to apply hex entity encoding to the payload, bypassing the WAF and enabling a UNION-based attack to extract credentials from the users table.

🎯 End Goals

  • Identify the XML-based stock check feature as a SQL injection point
  • Confirm that a WAF blocks direct UNION injection attempts
  • Bypass the WAF by encoding the payload using Hackvertor hex entity encoding
  • Use a UNION attack to extract usernames and passwords from the users table
  • Log in as administrator to solve the lab

🪜 Steps to Reproduce

  1. Browse the lab and observe the server assigns a TrackingId cookie on initial page load; confirm the browser sends it on subsequent requests
  2. Navigate to a product page and click “Check stock” — observe the application returns a stock count
  3. In Burp HTTP History, locate the POST /product/stock request and observe it sends an XML body with productId and storeId fields
  4. Send the request to Burp Repeater
  5. Inject 1 UNION SELECT NULL into the storeId field and observe a 500 error — confirming a WAF is blocking the payload
  6. Using the Hackvertor extension, select the UNION payload, right-click → Extensions → Hackvertor → Encode → base64; observe the response still fails
  7. Switch to hex entity encoding: select the payload, right-click → Extensions → Hackvertor → Encode → hex_entities; observe 200 OK — the WAF is bypassed
  8. Confirm the number of columns by trying UNION SELECT NULL,NULL — a 200 response with “0 units” confirms only 1 column is returned
  9. Inject UNION SELECT username||'~'||password FROM users (hex-entity encoded) to concatenate and retrieve all credentials in a single column
  10. Extract the administrator password from the response and log in to solve the lab

🧠 Analysis

Step 1 — Observe the lab and identify the TrackingId cookie

Browsing the lab homepage showed it is marked “Not solved”. In Burp HTTP History, the server issued a TrackingId cookie on the initial GET / request. Navigating to a product page confirmed the browser automatically sends this cookie on subsequent requests, meaning it is processed server-side on every page load.

Step 2 — Identify the stock check feature and capture the request

Clicking “Check stock” on a product page returned “258 units”. In Burp HTTP History, the corresponding POST /product/stock request was visible — it sends an XML body with productId and storeId values to a backend stock query. This XML structure is a non-standard injection surface that may not be covered by standard WAF rules. The request was sent to Burp Repeater for testing.

Step 3 — Confirm WAF blocks direct UNION injection

With the request in Burp Repeater, 1 UNION SELECT NULL was injected directly into the <storeId> field:

<storeId>1 UNION SELECT NULL</storeId>

The server responded with HTTP 500 Internal Server Error — confirming a WAF detected and blocked the plaintext SQL keywords.

Step 4 — Try base64 encoding (blocked), then switch to hex_entities (succeeds)

To bypass the WAF, the Hackvertor Burp extension was used to obfuscate the payload. The 1 UNION SELECT NULL value was first wrapped in a base64 Hackvertor tag:

<storeId><@base64>1 UNION SELECT NULL</@base64></storeId>

This did not work — the WAF still blocked it or the database did not decode the base64. Hex entity encoding was then tried instead, wrapping the payload in <@hex_entities> tags:

<storeId><@hex_entities>1 UNION SELECT NULL</@hex_entities></storeId>

The server returned 200 OK with null in the response body — confirming the WAF was bypassed. The XML parser decoded the hex entities before executing the query, so the database saw valid SQL while the WAF saw only encoded text.

Step 5 — Confirm single column

To determine the number of columns returned, UNION SELECT NULL,NULL was tested:

<storeId><@hex_entities>1 UNION SELECT NULL,NULL</@hex_entities></storeId>

The server returned 200 OK with “0 units” — confirming the query returns only 1 column, meaning the full credential string must be concatenated into a single value.

Step 6 — Exfiltrate credentials with username/password concatenation

With 1 column confirmed, usernames and passwords were concatenated using ||'~'|| as a separator and extracted in a single UNION query:

<storeId><@hex_entities>1 UNION SELECT username||'~'||password FROM users</@hex_entities></storeId>

The response returned all user credentials from the users table. The administrator’s credentials were clearly visible: t0vc3kip9bisx2bc6d5n

Step 7 — Log in as administrator and solve the lab

Logging in at /login with username administrator and the extracted password t0vc3kip9bisx2bc6d5n returned the “My Account” page showing “Your username is: administrator” and the “Congratulations, you solved the lab!” banner.

🎯 Impact

An attacker who can reach the stock check endpoint can extract arbitrary data from the database by bypassing the WAF using XML entity encoding — a technique that requires only the Hackvertor Burp extension and is trivial to apply. In this case, all user credentials were recovered from the users table in a single request, enabling complete account takeover. On a production system this technique could be used to exfiltrate any data accessible to the database user — including session tokens, API keys, PII, or any other sensitive records — while evading WAF detection entirely.
