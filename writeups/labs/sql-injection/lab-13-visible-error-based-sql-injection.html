Lab 13 — Visible error-based SQL injection

🟡 Medium 📅 2026-05-19

📋 Finding Summary

FieldValue
TypeWeb App
Severity🟡 Medium

🔍 Description

Visible error-based SQL injection occurs when a web application includes raw database error messages in HTTP responses. An attacker can craft payloads that cause the database to raise errors containing the output of arbitrary SQL expressions — effectively turning error messages into a data exfiltration channel. In this lab, the TrackingId cookie value is embedded unsanitized into a SQL query, and PostgreSQL error messages are reflected directly in the HTTP response, allowing the attacker to extract sensitive data one value at a time using CAST-based payloads.

🎯 End Goals

  • Confirm the TrackingId cookie is vulnerable to SQL injection via visible error messages
  • Identify the database type and version through error-based data leakage
  • Extract the administrator username and password from the users table
  • Log in as administrator to solve the lab

🪜 Steps to Reproduce

  1. Browse the lab and observe that the server assigns a TrackingId cookie to the browser
  2. Navigate to the login page and confirm the browser sends the TrackingId cookie on subsequent requests
  3. In Burp Repeater, inject a single quote into the TrackingId value and observe a 500 error with a visible SQL error message
  4. Add a SQL comment (-) after the quote to confirm the injection point produces a clean 200 response
  5. Use CASTbased payloads to probe boolean expression behavior, refining syntax until the server returns 200
  6. Inject CAST((SELECT version()) AS int) to leak the PostgreSQL version string through the error message
  7. Confirm the users table exists by injecting CAST((SELECT 'a' from users) AS int) without the TrackingId value to keep the query short
  8. Add LIMIT 1 and switch to username to extract the first username via the error message
  9. Repeat with password to extract the administrator’s password
  10. Log in using the extracted credentials to solve the lab

🧠 Analysis

Step 1 — Observe initial server behavior

Browsing the lab homepage in Burp showed that the server issues a TrackingId cookie (e.g. R9odV6qtSBZRwe8U) on the initial GET / request, returning 200 OK. Navigating to /login confirmed that the browser automatically sends this TrackingId on subsequent requests, meaning the cookie value is processed server-side on every page load — making it a candidate injection point.

Step 2 — Inject a single quote to trigger an error

A single quote was appended to the TrackingId value: TrackingId=R9odV6qtSBZRwe8U'. The server responded with HTTP 500 and reflected the full PostgreSQL error: Unterminated string literal started at position 52 in SQL SELECT \ FROM tracking WHERE id = ‘R9odV6qtSBZRwe8U’. Expected char*. This confirmed that the cookie value is embedded directly into a SQL query with no sanitization, and that database errors are visible in the response.

Step 3 — Comment out the trailing quote to confirm injection

Adding a SQL comment after the injected quote — TrackingId=R9odV6qtSBZRwe8U'-- — neutralized the syntax error by commenting out the application’s closing quote. The server returned 200 OK and the homepage loaded normally, confirming that the injection point is controllable and that payloads appended after the value are executed by the database.

Step 4 — Probe with CAST to establish a valid boolean expression

To use CAST for error-based extraction, the payload needs to form a valid boolean expression in the AND clause. Initial attempts showed two errors requiring correction:

First, ' AND CAST((SELECT(1)) as int)-- returned HTTP 500 with ERROR: syntax error at or near “SELECT” Position: 62 — SELECT(1) with parentheses directly around the literal value is invalid SQL syntax.

Correcting to ' AND CAST((SELECT 1) AS int)-- (without =1) still returned 500 with ERROR: argument of AND must be type boolean, not type integer — the CAST result needs to be compared to something to form a boolean.

Adding the comparison — ' AND CAST((SELECT 1) AS int)=1-- — returned 200 OK, confirming the full CAST(...) = value pattern works as a valid boolean expression in the AND clause and is ready for data extraction.

Step 5 — Leak the database version

With the CAST pattern confirmed, version() was substituted: ' AND CAST((SELECT version()) AS int)=1--. The server returned 500 with the error: ERROR: invalid input syntax for type integer: “PostgreSQL 12.22 (Ubuntu 12.22-0ubuntu0.20.04.4) on x86_64-pc-linux-gnu, compiled by gcc (Ubuntu 9.4.0-1ubuntu1~20.04.2) 9.4.0, 64-bit”. The database version string was fully leaked through the error message, confirming PostgreSQL as the backend.

Step 6 — Confirm the users table exists and extract the username

To keep the query short enough for the database to process cleanly, the TrackingId value was dropped, sending only 'AND CAST((SELECT 'a' from users) AS int)=1--. The server returned 500 with ERROR: more than one row returned by a subquery used as an expression, confirming the users table exists and contains multiple rows.

Adding LIMIT 1 and switching to the username column — 'AND CAST((SELECT username from users LIMIT 1) AS int)=1-- — returned the error: ERROR: invalid input syntax for type integer: “administrator”, leaking the first username through the error message.

Step 7 — Extract the password and log in

Substituting password for username — 'AND CAST((SELECT password from users LIMIT 1) AS int)=1-- — returned the error: ERROR: invalid input syntax for type integer: “qbdahza2n0r6yjsfzikf”, leaking the administrator’s password: qbdahza2n0r6yjsfzikf.

Logging in at /login with username administrator and the extracted password qbdahza2n0r6yjsfzikf returned the “Congratulations, you solved the lab!” banner.

🎯 Impact

An attacker with access to the TrackingId cookie can extract arbitrary data from the database one value at a time using CAST-based payloads. In this case, full account credentials were recovered from the users table, enabling complete account takeover. On a production system this technique could be used to exfiltrate usernames, password hashes, session tokens, or any other sensitive data accessible to the database user.
