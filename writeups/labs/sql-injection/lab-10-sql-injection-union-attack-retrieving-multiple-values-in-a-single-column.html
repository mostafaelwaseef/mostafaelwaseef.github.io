Lab 10 — SQL Injection UNION Attack — Retrieving Multiple Values in a Single Column

⚪ 🔴 High 📅 2026-05-12

📋 Finding Summary

FieldValue
TypeWeb App
Severity🔴 High
CVE / CWECWE-89 — SQL Injection
Date2026-05-12

🔍 Description

This lab demonstrates a UNION-based SQL injection vulnerability in a product category filter on a PostgreSQL database. The application returns query results in the response but only one column accepts string data, requiring the attacker to concatenate multiple values into a single output column. By probing column count, confirming which column accepts strings, identifying the database type, and leveraging PostgreSQL's information_schema to enumerate tables and columns, the attacker retrieves plaintext credentials and achieves full administrative account takeover.

🎯 End Goals

  • Determine the number of columns returned by the query
  • Identify which column accepts string data
  • Confirm the database type and version
  • Retrieve all usernames and passwords from the users table (columns username and password are known from the lab description)
  • Log in as the administrator user

🪜 Steps to Reproduce

  1. Intercept the product category filter request in Burp Suite
  2. Probe column count by incrementing NULLs in a UNION SELECT until a 200 OK is returned
  3. Identify which column accepts string data by substituting each NULL with a string literal
  4. Identify the database type by testing @@version (MSSQL/MySQL) then version() (PostgreSQL)
  5. Query information_schema.tables to list all table names and identify the users table
  6. Query information_schema.columns filtered on the users table to retrieve column names
  7. Dump credentials by concatenating username and password into the single string-compatible column
  8. Log in to the application as administrator

🧠 Analysis

Step 1 — Column count confirmation

The category filter parameter was injected with a UNION SELECT payload to determine how many columns the original query returns. A single NULL returned a 500 error, indicating an incorrect column count. Adding a second NULL returned a 200 OK, confirming the query returns exactly two columns.

' UNION SELECT NULL--
' UNION SELECT NULL,NULL--

Step 2 — String type confirmation

With two columns confirmed, each column was tested for string compatibility by substituting a string literal in place of NULL. Placing 'a' in the first column position returned a 500 error, ruling it out. Placing 'a' in the second column position returned a 200 OK, confirming only the second column accepts string data.

' UNION SELECT 'a',NULL--
' UNION SELECT NULL,'a'--

Step 3 — Database version confirmation

The database type was probed using @@version — a syntax specific to MSSQL and MySQL. This returned a 500 error, ruling out both database types. The PostgreSQL-specific version() function was then tried, which returned a 200 OK and rendered the version string in the browser, confirming the target runs PostgreSQL 12.22.

' UNION SELECT NULL,@@version--
' UNION SELECT NULL,version()--

Step 4 — Table enumeration via information_schema.tables

A Google search was performed to review the information_schema.tables view schema and identify which columns were available and relevant to use. The table_name column was identified as the target. The payload returned the full list of tables in the response. Searching the output revealed a table named users.

' UNION SELECT NULL,table_name FROM information_schema.tables--

Step 5 — Column enumeration via information_schema.columns

A Google search was performed to review the information_schema.columns view schema and identify which columns were available and relevant to use. The view was queried filtered on users. The response revealed three columns: email, password, and username.

' UNION SELECT NULL,column_name FROM information_schema.columns WHERE table_name = 'users'--

Step 6 — Dump credentials and login

Since only one column accepts string data, both username and password were concatenated into a single value using the || operator with a ~ separator. The browser rendered all credentials including administrator with its password. The administrator account was used to log in, solving the lab.

' UNION SELECT NULL,username || '~' || password FROM users--

🎯 Impact

An attacker with access to this SQL injection point can fully enumerate the database schema and extract any stored data including plaintext credentials. Even when the query returns only a single string-compatible column, concatenation techniques allow full credential extraction, enabling complete account takeover of any user including the administrator, and potentially lateral movement if credentials are reused elsewhere.
