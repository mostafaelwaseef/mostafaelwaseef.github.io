Lab 4 — LFI to RCE via Log Poisoning

🔴 Critical 📅 2026-07-23

📋 Finding Summary

FieldValue
TypeWeb App
Severity🔴 Critical
CVE / CWECWE-98, CWE-73
Date2026-07-23

🔍 Description

Level 4 presents a PHP application that passes user-supplied input directly into an include() statement without any filtering or sanitization. The page parameter in level4.php is appended to the string "includes/" and passed to PHP’s include() function, allowing an attacker to traverse the filesystem with ../ sequences and include arbitrary files. The server stores Apache access logs at a non-standard path (/var/www/html/logs/access.log), and the log file is world-readable. Because Apache logs the HTTP User-Agent header verbatim into the access log, an attacker can write a PHP webshell into the log by sending a crafted request with a malicious User-Agent value. Once the payload is written to the log, including the log file via the LFI vulnerability causes the PHP engine to execute the embedded code, achieving Remote Code Execution (RCE).

🎯 End Goals

  • Exploit the LFI vulnerability to read arbitrary files from the server
  • Discover the location of the Apache access log
  • Inject a PHP webshell into the log via a crafted User-Agent header (log poisoning)
  • Trigger the webshell by including the poisoned log via LFI
  • Retrieve the flag from /var/www/html/flags/flag4.txt

🪜 Steps to Reproduce

  1. Identify the vulnerable page parameter in level4.php that passes unsanitized input to include().
  2. Confirm LFI by using directory traversal sequences to read /etc/passwd.
  3. Enumerate accessible log file paths to find one readable via LFI.
  4. Verify the discovered log file is readable by including it through the LFI endpoint.
  5. Poison the log by sending a request with a PHP webshell in the User-Agent header.
  6. Trigger the webshell by including the log file via the LFI parameter and passing a command via the cmd query parameter.
  7. Confirm RCE by executing id and observing the output in the response.
  8. Use the webshell to locate the flag file and retrieve its contents.

🧠 Analysis

Step 1 — Explore the application

Navigating to level4.php with the default page=welcome.txt parameter reveals that the application loads files from an includes/ directory. The URL structure and the “Loading: includes/welcome.txt” message confirm that the page parameter is passed directly into a PHP include() call, making it a potential LFI entry point.

Step 2 — Attempt path inclusion without traversal

Submitting page=/etc/passwd without any traversal sequences triggers a PHP warning: include(includes//etc/passwd): failed to open stream. The error message exposes the full include path structure, confirming the application prepends includes/ to the parameter value and that direct absolute paths fail without traversal.

Step 3 — Confirm LFI with path traversal

Using four levels of directory traversal with page=../../../../etc/passwd successfully returns the contents of /etc/passwd. This confirms the LFI vulnerability is exploitable and that four ../ sequences are sufficient to escape the includes/ directory and reach the filesystem root.

Step 4 — Discover and verify the log path

Testing the standard Apache log path page=../../../../var/log/apache2/access.log returns a PHP error, confirming the log is not at the default location. A Python script is used to scan 12 common log paths, sending an LFI request for each and checking whether the response contains PHP include errors. The script identifies var/www/html/logs/access.log as the only accessible log file:

import requests

paths = [
    "var/log/apache2/access.log",
    "var/log/apache2/error.log",
    "var/log/nginx/access.log",
    "var/log/php_errors.log",
    "var/log/syslog",
    "var/log/auth.log",
    "proc/self/environ",
    "tmp/access.log",
    "var/www/html/logs/access.log",
    "var/www/logs/access.log",
    "var/log/apache/access.log",
    "usr/local/apache2/logs/access.log",
]
base = "http://localhost:8080/level4.php"
traversal = "../" * 4
for path in paths:
    r = requests.get(f"{base}?page={traversal}{path}")
    if r.status_code == 200 and "failed to open stream" not in r.text and "Failed opening" not in r.text:
        print(f"[FOUND] {path}")
    else:
        print(f"[MISS]  {path}")

Navigating to page=../../../../var/www/html/logs/access.log in the browser confirms the log is readable via LFI and displays all recorded HTTP requests, including User-Agent values.

Step 5 — Poison the log and confirm RCE

A PHP webshell is injected into the access log by sending a request with <?php system($_GET['cmd']); ?> as the User-Agent header using curl. Apache logs the header verbatim into /var/www/html/logs/access.log. Including the poisoned log via page=../logs/access.log&cmd=id causes PHP to execute the embedded code, returning uid=33(www-data) gid=33(www-data) groups=33(www-data) in the response, confirming RCE as the www-data user.

Step 6 — Locate and read the flag

With RCE confirmed, the command find / -name flag4.txt 2>/dev/null is sent via curl by appending &cmd=find/+-name+flag4.txt+2>/dev/null to the LFI URL and piping the output through grep flag4. The result reveals the flag at /var/www/html/flags/flag4.txt. The flag is then retrieved by navigating directly to the URL with cmd=cat+/var/www/html/flags/flag4.txt in the browser, which renders the log output inline: FLAG{lvl4_log_poisoning_rce}

🎯 Impact

An attacker with access to this vulnerability can achieve full Remote Code Execution on the web server as the www-data user. This allows reading of arbitrary files, exfiltration of sensitive data, modification of web application files, and potential lateral movement within the server environment or internal network.
