Level 3 — Forced Extension Bypass

🔴 High 📅 2026-07-21

📋 Finding Summary

FieldValue
TypeWeb App
Severity🔴 High
CVE / CWECWE-98, CWE-22, CWE-209
Date2026-07-21

🔍 Description

The application appends a hardcoded .php extension to every value supplied via the page parameter before passing it to PHP’s include(). The intent is to prevent inclusion of arbitrary file types — an attacker submitting ../../../../etc/passwd receives includes/../../../../etc/passwd.php, which does not exist. However, this defense backfires when the target file itself has a .php extension: the forced suffix becomes a match rather than a blocker. The flag was stored as flag3.php, so the payload ../flags/flag3 resolves to ../flags/flag3.php and is included successfully. Additionally, the application leaks its full server path through PHP error messages, giving an attacker precise information about the web root before any bypass attempt.

🎯 End Goals

  • Identify that .php is always appended to the user-supplied filename
  • Confirm the application leaks its server path through error disclosure
  • Recognize that the forced extension becomes an advantage when the target file is itself a .php file
  • Include flags/flag3.php via ../flags/flag3 to capture the flag

🪜 Steps to Reproduce

  1. Navigate to the target and observe the page parameter and Loading: path in the response
  2. Submit an absolute path probe to trigger a PHP error and confirm the web root and forced .php extension
  3. Verify LFI works on .php targets by self-including the level page
  4. Apply the insight to ../flags/flag3 — the app appends .php, which resolves flag3.php and returns the flag

🧠 Analysis

Step 1 — Observe the application

Navigating to http://localhost:8080/level3.php shows the familiar page-navigation interface. The default page=welcome loads with Loading: includes/welcome.php visible in the response — the application is explicitly appending .php to every parameter value. A hint in the page notes that null-byte truncation (%00) was patched in PHP 5.3.4 and will not work here.

Step 2 — Error disclosure confirms web root and forced extension

Submitting page=/etc/passwd triggered a PHP include() error. The Loading: line showed includes//etc/passwd.php — the .php suffix appended to /etc/passwd confirms the forced-extension behavior. More importantly, the PHP error message disclosed the full server path: /var/www/html/level3.php, establishing the web root as /var/www/html/. An attacker now knows both the directory layout and exactly why standard traversal to arbitrary file types will fail.

Step 3 — Self-inclusion confirms LFI works on .php files

To verify that the forced extension is an advantage rather than a blocker when the target is a PHP file, the payload ../level3 was submitted. The app appended .php, producing Loading: includes/../level3.php, which resolves to level3.php itself. The page rendered recursively — Level 3 including itself — confirming that LFI is fully functional as long as the target file carries a .php extension.

Step 4 — Retrieve the flag

With the mechanism understood, the payload ../flags/flag3 was submitted. The application appended .php, resolving to Loading: includes/../flags/flag3.php — pointing to /var/www/html/flags/flag3.php. The file was included and the flag returned: FLAG{lvl3_wrapper_bypass_forced_extension}

🎯 Impact

The forced-extension defense creates a false sense of security. It prevents reading non-PHP files (configuration files, /etc/passwd, text files) but leaves any .php file on the server fully readable via LFI. In a real application this means source code disclosure, exposure of files containing credentials or business logic, and potential for chained attacks. Combined with the PHP error-disclosure that leaks the web root, an attacker can enumerate the directory structure and target PHP files with high precision before making a single exploit attempt.
