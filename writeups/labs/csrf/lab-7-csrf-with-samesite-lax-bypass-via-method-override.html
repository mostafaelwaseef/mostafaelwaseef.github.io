Lab 7 — CSRF with SameSite Lax bypass via method override

🟡 Medium 📅 2026-06-25

📋 Finding Summary

FieldValue
TypeWeb App
Severity🟡 Medium
CVE / CWECWE-352
Date2026-06-25

🔍 Description

SameSite Lax is a browser cookie attribute intended to prevent CSRF by blocking cross-site POST requests from carrying session cookies. However, it permits cross-site GET requests that result from top-level navigations (e.g., clicking a link). If the server also supports HTTP method override — accepting a _method=POST query parameter to treat a GET request as a POST — an attacker can bypass the SameSite Lax restriction entirely. By crafting a GET request with _method=POST, the attacker’s forged request is treated as a POST server-side, while the browser still sends the SameSite=Lax session cookie because it originated from a top-level GET navigation.

🎯 End Goals

  • Identify that the email change endpoint relies on SameSite Lax cookie protection rather than a CSRF token
  • Discover that the server supports HTTP method override via _method=POST query parameter
  • Demonstrate that a GET request with _method=POST bypasses SameSite Lax and is processed as a POST
  • Craft a CSRF PoC that changes the victim’s email and deliver it through the exploit server

🪜 Steps to Reproduce

  1. Log in as wiener and submit an email change; capture the POST in Burp — note the absence of a CSRF token.
  2. Convert the POST to a GET in Burp Repeater — confirm 405 Method Not Allowed.
  3. Add _method=POST to the GET query string — confirm the server returns 302 Found and accepts the request.
  4. Right-click → Engagement tools → Generate CSRF PoC; build a GET-based form with _method=POST.
  5. Paste the PoC into the exploit server and deliver it to the victim.

🧠 Analysis

Step 1 — Access the lab

The lab was launched and the homepage loaded. The lab is Practitioner level and targets SameSite Lax cookie restrictions that can be bypassed via HTTP method override.

Step 2 — Log in, capture POST, and confirm baseline

Authenticated as wiener:peter and navigated to the My Account page, which showed the current email wiener@normal-user.net. A test email test1@test.com was entered and submitted. Burp Repeater captured the POST /my-account/change-email request — notably, there was no CSRF token in the request body. The application relies on the browser’s SameSite=Lax cookie policy to block cross-site POST requests rather than using token-based protection. The email was updated to test1@test.com, confirming the baseline flow.

Step 3 — Change method to GET, confirm 405, then bypass with _method=POST

Right-clicked the POST in Burp Repeater → Change request method, converting it to GET /my-account/change-email?email=.... The server returned HTTP/2 405 Method Not Allowed, confirming the endpoint normally rejects GET requests. The _method=POST parameter was then appended to the query string. The GET was resent — the server returned HTTP/2 302 Found and redirected to /my-account?id=wiener. The My Account page confirmed the email had been updated to test3@test.com. This proved the server honours the _method override, treating the GET as a POST internally, while SameSite Lax permits the GET because browsers allow top-level cross-site GET navigations to carry Lax cookies.

Step 4 — Generate CSRF PoC and paste into exploit server

Right-clicked the GET request in Burp Repeater → Engagement tools → Generate CSRF PoC. The generator built a GET-based HTML form with method="GET" targeting /my-account/change-email, with hidden fields for email=test4@test.com and _method=POST. The generated HTML was pasted into the exploit server body field.

Step 5 — Deliver exploit and solve

“Deliver exploit to victim” was clicked. The victim’s browser navigated to the exploit page, which triggered the GET form submission — a top-level navigation that SameSite Lax permits. The server received the _method=POST override and processed the request as a POST, accepting the email change. The lab banner updated to Solved, and the My Account page confirmed the email was changed to test4@test.com.

🎯 Impact

An attacker who can lure an authenticated user to visit a malicious page can silently change the victim’s email address even when SameSite Lax is in place — by exploiting the server’s HTTP method override support. Because SameSite Lax only blocks cross-site cookies on POST requests (and on sub-resource loads), a top-level GET navigation carrying _method=POST bypasses the restriction entirely. This allows silent state-changing actions such as email takeover, which can be chained with a password reset to achieve full account takeover.
