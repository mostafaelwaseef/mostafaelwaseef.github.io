Lab 8 — CSRF with SameSite Strict bypass via client-side redirect

⚪ 🟡 Medium 📅 2026-06-25

📋 Finding Summary

FieldValue
TypeWeb App
Severity🟡 Medium
CVE / CWECWE-352
Date2026-06-25

🔍 Description

SameSite=Strict prevents cross-site requests from carrying session cookies entirely — even top-level navigations. However, if the application performs a client-side redirect using attacker-controllable input that stays within the same origin, the browser treats the resulting navigation as same-site and includes the SameSite=Strict cookie. In this lab, the blog comment confirmation page reads a postId query parameter and uses it directly in a JavaScript window.location redirect: window.location = blogPath + '/' + postId. By supplying a path-traversal postId value (e.g. /../my-account/change-email?email=...&submit=1), the redirect can be pointed at any same-origin endpoint. When a victim visits the exploit page, the same-site redirect carries their session cookie, turning a client-side gadget into a full CSRF bypass.

🎯 End Goals

  • Identify that the session cookie uses SameSite=Strict, blocking conventional cross-site CSRF
  • Discover the comment confirmation page performs a client-side redirect using a controllable postId parameter
  • Demonstrate that a path-traversal postId value can redirect to the email change endpoint with the session cookie attached
  • Craft an exploit that triggers the redirect via document.location and deliver it to change the victim’s email

🪜 Steps to Reproduce

  1. Log in as wiener — note the session cookie has SameSite=Strict.
  2. Submit a comment on a blog post and observe the confirmation page redirects via JavaScript using the postId URL parameter.
  3. Inspect the redirect script — confirm postId is used unsanitised in window.location.
  4. Test a crafted postId path-traversal value that points to /my-account/change-email?email=...&submit=1.
  5. Confirm the GET-based email change endpoint accepts the request when triggered by same-site redirect.
  6. Build an exploit page using document.location pointing to the crafted confirmation URL and deliver it via the exploit server.

🧠 Analysis

Step 1 — Access the lab

The lab was launched and the homepage loaded. The lab is Practitioner level and targets a SameSite=Strict bypass via a client-side redirect gadget.

Step 2 — Log in and confirm SameSite=Strict cookie

Authenticated as wiener:peter. Burp captured the POST /login response, which set Cookie: session=...; Secure; HttpOnly; SameSite=Strict — confirming the application enforces the strictest SameSite policy. After login, the My Account page showed the current email wiener@normal-user.net.

Step 3 — Discover the comment confirmation redirect

Submitted a test comment on a blog post (comment: test, name: test, email: test@test.com). The server redirected to /post/comment/confirmation?postId=1, which displayed “Thank you for your comment! You will be redirected momentarily.” After a brief delay, the page navigated back to /post/1, confirming the presence of a JavaScript-driven redirect that uses the postId query parameter.

Step 4 — Inspect the redirect script and confirm postId injection

Captured the POST /post/comment in Burp Repeater and followed the chain through to the confirmation page. Inspected the confirmation page source — the redirectOnConfirmation JavaScript function reads postId directly from URLSearchParams and constructs the redirect target as window.location = blogPath + '/' + postId after a 3-second timeout. No sanitisation is applied to postId, meaning a value like /../my-account/change-email%3Femail%3D...%26submit%3D1 will resolve to a same-origin email change URL. Because the redirect stays on the same origin, the browser treats the navigation as same-site and includes the SameSite=Strict session cookie.

Step 5 — Test the redirect payload and confirm same-site navigation

Navigated directly to the confirmation page with a crafted postId value pointing to /../my-account/change-email%3Femail%3Dtest%40test.com%26submit%3D1. The confirmation page loaded, the URL bar showed the redirect in progress, and the page navigated to My Account. The redirect was confirmed to be same-site — the browser included the SameSite=Strict session cookie throughout the chain.

Step 6 — Capture email change POST and convert to GET

Submitted the email change form with test1@test.com to capture the POST to /my-account/change-email. In Burp Repeater, right-clicked → Change request method to convert to GET — the parameters moved into the query string as email=test1%40test.com&submit=1. This GET form of the request is what the redirect will trigger.

Step 7 — Confirm the full redirect chain changes the email

Navigated to the crafted confirmation URL with postId=/../my-account/change-email?email=test2@test.com&submit=1. The “Thank you” confirmation page appeared, the redirect fired after 3 seconds, and the My Account page confirmed the email had been updated to test2@test.com — proving the entire chain works: same-site redirect → GET email change → SameSite=Strict cookie included.

Step 8 — Build exploit and deliver to victim

Pasted the following payload into the exploit server body:

<script>
document.location = "https://LAB-ID.web-security-academy.net/post/comment/confirmation?postId=/../my-account/change-email?email=test3%40test.com%26submit=1";
</script>

Clicked “Deliver exploit to victim”. The victim’s browser navigated to the exploit page, which immediately redirected to the confirmation URL. The same-site redirect carried their SameSite=Strict session cookie to the email change endpoint. The lab banner updated to Solved, and the My Account page confirmed the email was changed to test3@test.com.

🎯 Impact

An attacker who can lure an authenticated victim to visit a malicious page can silently change their email address even when SameSite=Strict is enforced — by chaining through a client-side redirect gadget that stays within the same origin. Because the redirect is same-site, the browser attaches the session cookie throughout the chain, completely bypassing the SameSite=Strict protection. This attack requires no CSRF token and no cookie manipulation, making it difficult to detect. The resulting email takeover can be chained with a password reset for full account takeover.
