Objective

Identify how the File Inclusion lab loads content at High security and determine what filtering is applied to the page parameter.

Findings

The File Inclusion lab page at High security presented the same three links — file1.php, file2.php, and file3.php. Clicking through each link confirmed that the page GET parameter continued to cycle through ?page=file1.php, ?page=file2.php, and ?page=file3.php. This indicated the parameter still controls server-side file inclusion, but that the High security level applies stricter filtering than previous levels.

PoC:
- Tools used: Browser
- Commands: Observed URL changing to ?page=file1.php, ?page=file2.php, ?page=file3.php by clicking each link
- Result: page parameter confirmed to still control server-side file inclusion at High security

Exploitation — Local File Inclusion (LFI) via PHP Stream Wrapper

Description

At High security, DVWA applies a fnmatch("file*", $file) check that requires the page parameter value to begin with the string "file". This blocks absolute paths (e.g. /etc/passwd) and path traversal sequences (e.g. ../../../../etc/passwd) because neither starts with "file". URL encoding and double URL encoding of traversal sequences are also blocked by the same check, since the raw string is matched before any decoding of path components.

The bypass leverages PHP’s file:// stream wrapper. The value file:///etc/passwd satisfies the fnmatch("file*", ...) check — it starts with "file" — and PHP’s include() function natively handles the file:// URI scheme, treating it as a direct local filesystem path. This allows reading arbitrary files without traversal sequences or absolute paths in the conventional sense.

PoC — Steps to Reproduce

  1. Observe the page parameter in the URL and attempt an absolute path: ?page=/etc/passwd — the application returns “ERROR: File not found!” confirming the filter is active
  2. Attempt standard path traversal: ?page=../../../../etc/passwd — also blocked with the same error, as the value does not begin with "file"
  3. Attempt URL-encoded traversal (?page=....%2f%2f....%2f%2fetc/passwd) and double URL-encoded traversal (?page=%252e%252e...%252fetc/passwd) — both blocked
  4. Bypass the fnmatch("file*", ...) filter using the PHP file:// stream wrapper, which satisfies the prefix check while still resolving to an arbitrary local path: ?page=file:///etc/passwd
  5. The server passes the value to include(), which resolves the file:// URI and renders the full contents of /etc/passwd in the browser

Lessons Learned

At High security, DVWA’s fnmatch("file*", $file) filter demonstrates a common weakness in pattern-based input validation: filtering for a prefix without accounting for URI schemes that satisfy the prefix while still reaching sensitive files. The file:// stream wrapper bypasses the check because it literally starts with "file", yet PHP’s include() resolves it as a local filesystem URI. This highlights that blacklist and pattern-matching approaches are fundamentally fragile.
